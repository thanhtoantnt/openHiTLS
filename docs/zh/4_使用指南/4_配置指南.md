# 特性及优化配置指南

openHiTLS架构高度模块化，RAM/ROM尺寸取决于所选的特性及优化配置。

## 1. 特性配置

openHiTLS 使用 CMake `option()` 变量来控制构建哪些特性。用户可以使用内置的预设（Preset），也可以通过 `-D` 选项对单个特性进行配置。

### 使用预设

使用 `-DHITLS_BUILD_PROFILE=<预设名>` 加载内置预设：

| 预设名 | 说明 |
|---|---|
| `full` | 开启所有特性 |
| `iso19790` | 符合 ISO/IEC 19790 标准的密码学算法库 |

也可以通过 CMake 的 `-C` 参数直接加载预设文件：

```bash
cmake .. -C ../cmake/presets/full.cmake
```

> **注意：** 通过 `-C` 加载的自定义预设文件必须包含 `set(HITLS_PRESET_LOADED ON CACHE BOOL "" FORCE)`。

### 组件与算法级别的开关

在选定预设的基础上，可使用 `-D` 开关对具体组件和算法进行微调：

- **组件级别**：`-DHITLS_CRYPTO=ON/OFF`、`-DHITLS_TLS=ON/OFF`、`-DHITLS_PKI=ON/OFF`、`-DHITLS_BSL=ON/OFF`
- **算法级别**：`-DHITLS_CRYPTO_<算法名>=ON/OFF`（例如 `-DHITLS_CRYPTO_AES=ON`、`-DHITLS_CRYPTO_SHA256=ON`）

启用某个特性时，构建系统会自动开启其所有依赖特性。

> **提示：** 所有可用的特性开关均定义在 [`cmake/hitls_options.cmake`](../../../cmake/hitls_options.cmake) 中，可通过查阅该文件了解全量可配置项。

> **注意：** 如果您在二次运行cmake时发现您主动关闭的特性还在开启状态，那可能有以下原因：
1. 上一次该特性开关开启了它下面的所有子特性，比如使用`cmake .. -DHITLS_CRYPTO_MD=ON`开启了所有的哈希算法子特性（如 SHA256、SHA512 等），即使您在二次运行 cmake 时尝试关闭 `HITLS_CRYPTO_MD`，由于其子特性仍然被开启，CMake 会根据依赖关系自动将其重新开启，从而导致您看到的特性仍然处于开启状态。这种行为是预期的，因为我们认为您使用`-DHITLS_CRYPTO_MD=ON`时就是希望开启其所有子特性。这时候如果您想关闭它的子特性，需要在命令行中显式关闭每一个子特性，例如 `-DHITLS_CRYPTO_SHA256=OFF -DHITLS_CRYPTO_SHA512=OFF`。
2. 您主动关闭的特性被其他特性强依赖。例如，如果您关闭了 `HITLS_CRYPTO_SHA256`，但同时开启了 `HITLS_CRYPTO_SCRYPT`，而 `HITLS_CRYPTO_SCRYPT` 依赖 `HITLS_CRYPTO_SHA256`，那么 CMake 会根据依赖关系自动将 `HITLS_CRYPTO_SHA256` 重新开启，从而导致您看到的特性仍然处于开启状态。这种行为也是预期的，因为构建系统会确保所有依赖关系得到满足。

### 特性配置示例

1. 开启 HPKE 算法：

   ```bash
   cmake .. -DHITLS_CRYPTO_HPKE=ON \
            -DHITLS_CRYPTO_HKDF=ON \
            -DHITLS_CRYPTO_SHA256=ON \
            -DHITLS_CRYPTO_AES=ON \
            -DHITLS_CRYPTO_GCM=ON \
            -DHITLS_CRYPTO_X25519=ON
   ```

2. 使用 TLS 1.3 协议，算法套件为 AES-128-GCM-SHA256：

   ```bash
   cmake .. -DHITLS_TLS_PROTO_TLS13=ON \
            -DHITLS_TLS_SUITE_AES_128_GCM_SHA256=ON \
            -DHITLS_CRYPTO_CURVE_NISTP256=ON \
            -DHITLS_TLS_SUITE_AUTH_ECDSA=ON
   ```

## 2. 优化配置

### 配置类别

#### 系统相关配置
|配置|说明|建议|
|---|---|---|
|HITLS_BSL_SAL_LINUX|使用Linux系统抽象层。用于适配Linux操作系统的系统调用。构建系统会自动从宿主机 OS 检测。|通常自动检测；交叉编译时按需指定|
|HITLS_CRYPTO_AUXVAL|使用辅助向量`getauxval`获取CPU特性。需要使用其他方式检测CPU特性。默认启用，会根据平台支持情况自动选择。|通常自动检测；根据需要选择|
|HITLS_CRYPTO_ASM_CHECK|启用汇编代码检查。在运行时检查CPU是否支持相应的指令集扩展。在汇编特性开启时会自动开启。|通常自动检测；根据需要选择|

#### 密钥生成优化配置
|配置|说明|建议|
|---|---|---|
|CRYPT_DH_TRY_CNT_MAX|DH密钥对生成的最大尝试次数，默认100。当生成的密钥不满足要求时,会重新生成,直到达到此次数限制。|保持默认值100，除非有特殊性能要求。|
|CRYPT_DSA_TRY_MAX_CNT|DSA密钥对生成的最大尝试次数，默认100。当生成的密钥不满足要求时,会重新生成,直到达到此次数限制。|保持默认值100，除非有特殊性能要求。|
|CRYPT_ECC_TRY_MAX_CNT|ECC密钥对生成的最大尝试次数，默认100。当生成的密钥不满足要求时,会重新生成,直到达到此次数限制。|保持默认值100，除非有特殊性能要求。|

#### ECC优化配置
|配置|说明|建议|
|---|---|---|
|HITLS_CRYPTO_NIST_ECC_ACCELERATE|使用NIST曲线的硬件加速。默认开启，可通过 `-DHITLS_CRYPTO_NIST_ECC_ACCELERATE=OFF` 关闭。此加速依赖INT128，若系统不支持，则忽略此配置。|默认开启|

#### 随机数生成配置
|配置|说明|建议|
|---|---|---|
|DRBG_MAX_RESEED_INTERVAL|DRBG(确定性随机数生成器)重新播种的最大间隔，默认10000。每生成10000次随机数后,需要重新获取熵源进行播种。|保持默认值10000，过大会降低随机数安全性，过小会影响性能。|
|HITLS_CRYPTO_ENTROPY_DEVRANDOM|使用操作系统提供的设备随机数作为熵源。在Linux系统上通常使用/dev/random或/dev/urandom。|若支持，则启用。|
|HITLS_CRYPTO_INIT_RAND_ALG|DRBG的初始化随机数算法。|默认值CRYPT_RAND_SHA256，可选值参考头文件`include/crypto/crypt_algid.h`中的CRYPT_RAND_AlgId|

#### 其他配置
|配置|说明|建议|
|---|---|---|
|HITLS_BSL_LOG_NO_FORMAT_STRING|日志输出不使用格式化字符串,直接输出原始字符串。可以提高日志性能。此功能主要被应用于协议模块。|如无需查看日志，则启用|
|HITLS_EAL_INIT_OPTS=n|EAL初始化选项。默认值为0，表示不启用EAL初始化。<br>当定义了 HITLS_EAL_INIT_OPTS 时，CRYPT_EAL_Init 和 CRYPT_EAL_Cleanup 会被标记为构造函数和析构函数，且会覆盖这两个函数的参数。<br>可设置不同的值，以启用不同的EAL初始化：<br>- CPU特性检测：CRYPT_EAL_INIT_CPU       0x01<br>- 错误码模块初始化：CRYPT_EAL_INIT_BSL       0x02<br>- 随机数初始化：CRYPT_EAL_INIT_RAND      0x04<br>- provider初始化：CRYPT_EAL_INIT_PROVIDER  0x08<br>n的值为上述值的和|根据需求启用|

### 配置方法

可参考[1_构建及安装指导](1_构建及安装指导.md)，在 CMake 配置阶段通过 `-D` 标志直接传入以下宏：

```bash
# 启用汇编检查，关闭 NIST ECC 硬件加速
cmake .. -DHITLS_CRYPTO_ASM_CHECK=ON -DHITLS_CRYPTO_NIST_ECC_ACCELERATE=OFF

# 启用特定的 EAL 初始化选项（9 = CRYPT_EAL_INIT_CPU + CRYPT_EAL_INIT_PROVIDER）
cmake .. -DHITLS_EAL_INIT_OPTS=9
```
